# 홀로라도 콘텐츠 작성 가이드

> 이 문서는 AI에게 글 작성을 요청할 때 함께 전달하는 기준 문서입니다.
> 아래 규칙을 지켜 홀로라도(holorado.me)의 블로그 글을 작성하세요.

---

## 1. 홀로라도는 어떤 곳인가

- **정체성**: "홀로여도 잘 산다." 1인가구의 삶을 담백하고 자족적으로 다루는 라이프스타일 블로그.
- **다루는 주제**: ① 1인가구 생활 ② 요리 ③ 생산성·AI
- **독자**: 혼자 사는 20~40대. 실용적인 해답을 찾아 검색으로 유입됨.
- **금지 정서**: 외로움 강조, 처량함, 신세 한탄, "혼자라서 힘들죠" 류의 동정 어조. 혼자 사는 삶은 **결핍이 아니라 잘 정돈된 여유**로 그린다.

## 2. 문체 (톤앤매너)

- **경어체 "~합니다/~해요" 혼용.** 설명은 "~합니다", 다가가는 문장은 "~해요/~죠"로 리듬을 준다.
  - 예: "핵심은 순서예요. 좋은 반찬통을 사는 것보다, 먼저 먹을 것을 눈앞에 두는 습관이 낭비를 훨씬 크게 줄입니다."
- **경험자가 담담히 알려주는 어조.** 전문가연하지 않고, 실제 해본 사람이 옆에서 말해주듯.
  - 예: "몇 번의 시행착오 끝에 낭비를 확실히 줄인 방법을 정리했습니다."
- **단정적이되 과장 없이.** "무조건", "최고의", "인생을 바꾸는" 같은 과장·낚시 표현 금지.
- **독자를 낮잡지 않기.** "이것도 모르셨죠?" 류 금지. 흔한 오해를 다룰 땐 "가장 흔한 오해입니다"처럼 현상을 지적한다.
- 이모지는 **글 전체에서 0~2개**, 마무리 문단이나 제목 옆 정도만. 본문 남발 금지.
- 문장은 짧게. 한 문장에 하나의 내용. 만연체 금지.

## 3. 글의 구조 (필수 형식)

모든 글은 아래 골격을 따른다:

```
[도입 2~4문장]  ← 독자가 겪는 구체적 상황/불편으로 시작. 공감 유도.
               ← 마지막 문장에서 "이 글이 주는 것"을 예고.

## [왜/원리 섹션]  ← 본론 첫 섹션은 "왜 이 문제가 생기는가" 원리 설명.
                   ← 원리를 알아야 팁이 납득되기 때문. 1~2문단.

## [실행 섹션 × 3~6개]  ← 팁/단계/방법. 각 섹션은:
   - 소제목은 행동이나 결론이 보이게 ("팁 1 — 껐다 켰다보다 켜두는 게 쌀 때가 많다")
   - 설명 문단 + 필요 시 불릿/표
   - 구체적 수치·예시 포함 ("1~2시간 이내 외출", "계란 2개 + 물 1.5배")

## 정리  ← 마지막 섹션 제목은 "정리" 또는 "마무리".
        ← 핵심을 1~2문장으로 압축 + 따뜻한 마무리 한 문장.
```

- 분량: **공백 포함 2,500~4,500자** (한국어 기준). 너무 짧으면 검색 가치가 없고, 너무 길면 늘어진다.
- 소제목(##)은 4~7개. h3(###)은 꼭 필요할 때만.
- 불릿은 3~6개 단위로. 불릿 하나가 두 줄을 넘지 않게.
- 표는 "비교"가 필요할 때만 (예: 압축 강도별 특징, 재료별 보관 기간).
- 인용구(>)는 글당 0~2개, 보조 팁("팁: ~") 용도로만.

## 4. 내용의 기준

- **검색 의도에 답하기.** 제목이 약속한 것을 본문이 실제로 해결해야 한다. 제목이 "전기세 줄이기"면 본문에 실행 가능한 절약 방법이 있어야 함.
- **구체성이 생명.** "적절히", "잘", "알아서" 금지. 숫자, 시간, 순서, 재료명으로 말한다.
- **직접 해본 것처럼 쓰되, 사실만.** 확인되지 않은 통계·수치·연구 인용 금지. 불확실하면 "~일 수 있습니다"로 표현 수위를 낮춘다.
- **한 글에 하나의 문제만.** 여러 주제를 욕심내지 않는다. 넓어질 것 같으면 글을 쪼갠다.
- **자연스러운 내부 연결.** 관련된 기존 글이나 도구가 있으면 본문 흐름 속에서 1회만, 억지 없이 링크. (예: "PDF가 무거워 첨부가 안 될 때는 [OptiPDF](https://pdf.holorado.me)로 줄이면 수월합니다.")
  - 도구 홍보가 글의 목적이 되면 안 된다. 글이 먼저, 도구는 곁들임.
- **앱·사이트·서비스를 소개하면 바로가기를 함께 제공한다.** 독자가 이름을 다시 검색하게 만들지 않는다.
  - 본문 안에 버튼 형태로 넣는다 (마크다운 안에 HTML 그대로 사용):
    ```html
    <a class="link-btn" href="https://공식주소" target="_blank" rel="noopener">서비스명 바로가기</a>
    ```
  - 버튼은 해당 서비스를 설명한 문단 바로 아래에, 글 전체에서 꼭 필요한 곳에만 (남발 금지).
  - **URL은 반드시 공식 도메인만.** 정부·공공기관은 `*.go.kr`/`*.or.kr`/기관 공식 도메인, 기업 서비스는 그 회사의 공식 사이트. 블로그·앱 소개 페이지·단축 URL을 대신 걸지 않는다.
  - AI가 URL을 지어내는 것도 흔한 할루시네이션이다. **버튼에 넣을 주소는 8번 검증 대상** — 실제 접속되는 공식 주소인지 확인 전엔 넣지 않는다. 확인 못 하면 버튼 없이 서비스명만 언급한다.
- **근거가 필요한 주장에는 출처를 본문에 링크로 표시한다.** 제도·요금·건강 등 🔴 주제의 주장에는 출처 주체를 언급하고, 검증된 공식 페이지를 링크로 건다.
  - 표기 형식(문장 흐름 속에 자연스럽게):
    - `...누진제가 적용됩니다 (출처: [한전 주택용 전기요금 안내](공식링크)).`
    - 또는 `[식약처 보관 기준](공식링크)에 따르면 ...`
  - 출처가 2개 이상이면 글 맨 아래(마무리 섹션 다음)에 `## 참고 자료` 섹션으로 모아 목록화해도 된다.
  - **어떤 주장에 출처가 필요한가**: 🔴(제도·요금·건강·안전·서비스 세부)만 출처 링크를 단다. 🟡(효과 크기·조리 시간 등 경험적 일반론)은 출처 대신 완화 표현("~인 경우가 많습니다")으로 충분하며 억지로 자리표시를 달지 않는다. 🟢(보편 상식)은 출처 불요.
  - **링크는 사람이 검증 후에만 삽입한다.** AI에게 글을 시킬 때는 링크를 직접 넣지 말고 자리표시로 남기게 한다: `[출처확인: 식약처 식품 보관 기준]` — 발행 전에 사람이 공식 페이지를 확인해 실제 링크로 교체하거나, 확인 못 하면 해당 주장을 완화/삭제한다.
  - ⚠️ **자리표시를 남긴 것은 "출처 확인"이 아니다.** `[출처확인: ...]` 은 "사람이 나중에 확인할 자리"라는 뜻이지, 확인이 끝났다는 뜻이 절대 아니다. AI가 "자리표시 완료"라고 보고해도 검증은 0% 진행된 상태다. **발행 전 본문에 `[출처확인:` 문자열이 하나라도 남아 있으면 미완성**이다 — 반드시 실제 링크로 교체하거나 그 주장을 뺀다.
  - 출처를 못 대는 🔴 주장은 쓰지 않는다 (8번 원칙).

## 5. 제목·설명 작성법

- **제목 형식**: `[핵심 키워드] — [구체적 약속]`
  - 예: "1인가구 여름 전기세 줄이기 — 에어컨 실사용 팁"
  - 예: "불 안 쓰는 여름 자취 요리 — 30분 안에"
  - 검색어가 제목 앞부분에 오게 한다. 낚시성 표현("충격", "꿀팁 대방출") 금지.
- **description(한 줄 설명)**: 2~3문장, 80~120자. ① 독자의 상황 공감 ② 이 글이 주는 해답 예고. 검색 결과와 카드에 그대로 노출됨.
  - 예: "혼자 살수록 여름 전기요금이 부담입니다. 에어컨을 끄는 게 답이 아니라 '어떻게 켜느냐'가 핵심이에요. 실제로 요금을 줄인 사용법을 정리했습니다."

## 6. 프론트매터 (파일 형식)

마크다운 파일 최상단에 아래 형식 그대로:

```yaml
---
title: "제목"
description: "한 줄 설명 (5번 기준)"
pubDate: 2026-07-06        # 발행일, YYYY-MM-DD
category: "1인가구"         # 셋 중 하나: 1인가구 | 요리 | 생산성·AI
heroEmoji: "🧊"             # 글을 대표하는 이모지 1개
draft: false
---
```

- category는 반드시 `1인가구`, `요리`, `생산성·AI` 중 하나. 다른 값 금지.
- heroEmoji는 주제를 직관적으로 보여주는 것 1개 (❄️ 전기세, 🥗 여름요리, 🤖 AI, 🧊 냉장고 등).

## 7. 하지 말 것 (금지 목록)

- ❌ 상투적 도입 전반: "안녕하세요", "오늘은 ~에 대해 알아보겠습니다/소개합니다/정리해봅니다", "이번 글에서는" 등 — **"오늘은/이번 글에서는"으로 시작하는 문장 자체를 금지.** 도입은 독자의 상황 묘사로 시작한다
- ❌ "지금까지 ~에 대해 알아보았습니다" 류의 상투적 마무리
- ❌ 과장·낚시: "충격", "무조건", "인생템", "미쳤다", "꿀팁 대방출"
- ❌ 외로움·동정 프레임: "혼자라서 서럽죠", "혼밥의 설움"
- ❌ 검증 불가한 수치·연구 인용 ("연구에 따르면 90%가~")
- ❌ 같은 접속사 반복 (그리고, 또한, 하지만 연달아 사용)
- ❌ 본문 이모지 남발, 느낌표 남발 (느낌표는 글 전체 0~3개)
- ❌ 목록만 나열하고 설명 없는 글 (불릿 아래엔 맥락 문장이 있어야 함)
- ❌ AI 티 나는 표현: "~라고 할 수 있습니다", "~하는 것이 중요합니다"의 기계적 반복

## 8. 할루시네이션(허위 정보) 검토 — 발행 전 필수

AI가 쓴 글은 **그럴듯하게 틀린 정보**를 섞을 수 있다. 홀로라도는 실용 정보 블로그라, 틀린 팁 하나가 독자에게 실제 손해(요금·건강·안전)를 줄 수 있고 블로그 신뢰 전체를 무너뜨린다. 발행 전 아래 검토를 반드시 거친다.

### 8-1. 위험도별 검토 기준

| 위험도 | 해당 내용 | 처리 |
|---|---|---|
| 🔴 높음 | 요금제·법·제도·정책 (누진제 구간, 지원금, 공휴일 지정 등), 건강·안전 (식품 보관 기간, 상한 음식 판별, 전기·화기 취급), 특정 서비스·앱의 기능과 가격 | **반드시 공식 출처로 확인** 후 기재. 확인 못 하면 구체 수치를 빼고 원리만 쓰거나 해당 내용 삭제 |
| 🟡 중간 | 일반적 생활 요령의 효과 크기 ("~하면 30% 절약"), 조리 시간·비율 | 수치를 단정하지 말 것. "체감이 꽤 달라집니다", "약 1분 → 상태 보며 30초씩" 처럼 폭·과정으로 표현 |
| 🟢 낮음 | 보편적 상식 수준의 원리 (더운 공기는 위로, 이미지가 파일을 무겁게 함) | 통상 검토 불요. 단 단정 어조는 원리가 확실할 때만 |

### 8-2. AI 글에서 특히 의심할 패턴

- **구체적인 숫자·퍼센트·연도**: "전기요금이 평균 23% 절감", "2024년 개정된 ~법" → 출처 없으면 전부 의심. AI가 가장 자주 지어내는 유형.
- **기관·연구·통계 인용**: "한국전력에 따르면", "연구 결과 ~%" → 실제 존재하는 발표인지 확인 전엔 쓰지 않는다. 가이드 원칙상 검증 불가 인용은 애초에 금지.
- **제품·앱·서비스의 세부 사항**: 이름, 기능, 가격, 무료 여부 → 실존하는지, 지금도 그런지 확인. 단종·유료화된 것을 추천하면 신뢰 타격.
- **날짜·시의성 정보**: 공휴일, 시즌 일정, 제도 시행일 → 연도가 걸린 내용은 반드시 당해 기준으로 확인.
- **URL·링크 주소**: AI가 그럴듯한 가짜 주소를 만들거나, 존재하지 않는 페이지 경로를 지어내는 경우가 흔하다. 본문에 넣는 모든 링크(특히 바로가기 버튼)는 실제 접속해 공식 도메인인지 확인한다.
- **만능 단정**: "모든 인버터 에어컨은~", "어떤 경우에도~" → 예외가 있는 표현은 "대부분", "요즘 제품 기준" 으로 완화.

### 8-3. 검증 방법

1. 글에서 **팩트성 문장에 전부 밑줄** 긋는다는 느낌으로 추려낸다 (숫자, 고유명사, 제도, 기한, 효과 주장).
2. 🔴 항목은 공식 출처(정부·공공기관 사이트, 해당 서비스 공식 페이지)에서 직접 확인한다. 블로그·커뮤니티 글은 출처로 인정하지 않는다.
3. 확인이 안 되는 내용의 처리 우선순위: **① 삭제 ② 수치 제거하고 원리만 ③ "~일 수 있습니다"로 완화**. "아마 맞겠지"로 발행하지 않는다. 반대로 **공식 출처로 확인된 🔴 주장은 그 출처를 본문에 링크로 남긴다** (4번 표기 형식) — 검증의 결과물을 독자와 공유하는 것이 신뢰의 근거가 된다.
4. AI에게 재검토를 시킬 땐 이렇게 요청한다: *"방금 쓴 글에서 사실 확인이 필요한 주장을 전부 목록으로 뽑고, 각각 확신도(확실/불확실)를 표시해줘."* — 불확실 표시가 붙은 항목만 골라 검증하면 효율적이다.
5. **AI의 확신도와 '근거'는 참고일 뿐, 검증이 아니다.**
   - 모든 항목에 "확실"이 붙어 나오면 오히려 의심한다. AI는 자기 글에 후한 판정을 내리는 경향이 있다.
   - 목록의 근거 칸에 적힌 기관명·문서명("○○청 가이드라인", "국가표준○○표")도 **AI가 지어낼 수 있다.** 근거 문구가 그럴듯하다고 검증된 것이 아니다. 🔴 항목은 AI의 확신도와 무관하게 사람이 직접 공식 출처를 확인한다.
   - 근거로 제시된 기관·문서가 실존하는지 애매하면, 그 근거는 없는 것으로 취급한다.

### 8-4. 표현 수위 원칙

- 검증한 사실 → 단정 가능: "누진제는 사용량 구간에 따라 단가가 올라갑니다."
- 검증 못 한 일반론 → 완화: "제품에 따라 다를 수 있습니다.", "~인 경우가 많습니다."
- 개인 경험 기반 → 경험임을 드러내기: "실제로 해보니 ~였습니다." (경험 프레임은 틀려도 거짓말이 아니다)
